# プロンプト③：実行予算画面の統合・簡素化

## 背景

Task #48（一括発注）と Task #54（仕入先入力マスタ選択型化）で本体機能は揃いましたが、現在「実行予算」の編集画面が2箇所に並立しており、UX上の混乱と保守コストの増加要因になっています。

### 現在の状態（Before）

```
プロジェクト詳細ページ（/projects/:id）
├─ 金額カード（請負金額・実行予算・実績原価・予想粗利）
└─ タブ切り替え
    └─ 「実行予算」タブ
        ├─ 集計カード（請負金額合計・当初予算合計・実行予算合計・予定利益率）
        ├─ 実行予算明細テーブル
        │   ├─ チェックボックス列（一括発注用）
        │   ├─ 行ごとの鉛筆アイコン → インライン編集モード（マスタ選択型ドロップダウン）
        │   ├─ 行ごとの削除アイコン
        │   └─ 「発注済」バッジ表示
        ├─ 「発注書を作成（N行選択中 / N社）」ボタン
        ├─ 「実行予算入力画面を開く」ボタン → 別ページへ遷移
        └─ 「行追加」ボタン

別ページ：実行予算入力画面（/projects/:id/budgets）
└─ グリッド編集UI
    ├─ 仕入先選択ドロップダウン（マスタ選択型・検索機能・クイック登録）
    ├─ 数量・単価などのインライン編集
    └─ 「行追加」「行削除」「見積書から取込」「保存する」ボタン
```

**問題点：**
- 2つの編集画面が並立し、どちらで何をするか不明瞭
- 同じドロップダウン UI が2箇所に重複実装（保守負担）
- タブ内の小さなテーブルでのインライン編集は操作性が低い
- 鉛筆アイコン→行編集モード→保存ボタンクリック、という多ステップ操作

### 目指す状態（After）

```
プロジェクト詳細ページ（/projects/:id）
├─ 金額カード（請負金額・実行予算・実績原価・予想粗利）
└─ タブ切り替え
    └─ 「実行予算」タブ
        ├─ 集計カード（請負金額合計・当初予算合計・実行予算合計・予定利益率）
        └─ 「実行予算を編集する」ボタン → 別ページへ遷移
        ※ 明細テーブル・チェックボックス・編集機能はすべて削除

実行予算入力画面（/projects/:id/budgets）← 唯一の編集画面
└─ グリッド編集UI（既存）
    ├─ 仕入先選択ドロップダウン（既存）
    ├─ 数量・単価などのインライン編集（既存）
    ├─ 「行追加」「行削除」「見積書から取込」「保存する」ボタン（既存）
    ├─ チェックボックス列【新規追加：タブから移植】
    ├─ 「発注書を作成（N行選択中 / N社）」ボタン【新規追加】
    ├─ 一括発注モーダル【既存ロジック流用】
    └─ 「発注済」バッジ列【新規追加】
```

**改善ポイント：**
- 編集画面が1箇所に統一され、ユーザーが迷わない
- グリッド編集UIに一括発注機能を統合し、業務フローがシームレスに
- 鉛筆アイコン経由の行編集モードは廃止（インライン編集で代替）
- コード重複が解消され、保守性向上

## 依頼内容

### ステップ1：事前調査（コード変更前に必ず実施）

以下を調査して報告してください：

1. **「実行予算タブ」の現状コンポーネント構造を特定**
   - 対象ファイル: プロジェクト詳細ページ（`/projects/:id`）の「実行予算」タブ部分
   - チェックボックス、「発注書を作成」ボタン、一括発注モーダル、「発注済」バッジ、鉛筆アイコン経由の行編集モード、削除ボタンの実装場所

2. **「実行予算入力画面」の現状コンポーネント構造を特定**
   - 対象ファイル: `/projects/:id/budgets` ページのコンポーネント
   - グリッド編集 UI のテーブル構造
   - 既存の各列・各ボタンの実装場所

3. **共有可能なロジック・コンポーネントの特定**
   - 一括発注モーダル本体（仕入先別グルーピング・納期/備考入力・確定処理）
   - 「発注書を作成」ボタンと選択カウンター
   - チェックボックスの非活性判定ロジック（vendor_id ベース）
   - 「発注済」バッジコンポーネント

4. **タブ側削除によって他に影響が出ないかの確認**
   - 「実行予算タブ」内で他のタブ（基本情報・原価明細・収支状況）と共有しているロジックがないか
   - 集計カードの計算ロジックが明細テーブルに依存していないか
   - URL ルーティング（タブ切替時のURL変化など）

調査結果を**コードスニペットと現状サマリー**で報告してください。実装はその後で。

### ステップ2：修正方針の提示

調査結果を踏まえて、以下を提示してください：

1. **「実行予算タブ」の簡素化方針**
   - 削除する要素のリスト（明細テーブル、チェックボックス、発注書作成ボタン、一括発注モーダル呼び出し、鉛筆アイコン経由の行編集モード、削除ボタン、行追加ボタン、「発注済」バッジ）
   - 残す要素（集計カードと、新規追加する「実行予算を編集する」ボタン）
   - 新規ボタンの遷移先：`/projects/:id/budgets`

2. **「実行予算入力画面」への機能統合方針**
   - チェックボックス列の追加位置（テーブル先頭が自然）
   - 「発注書を作成（N行選択中 / N社）」ボタンの配置（既存の「保存する」ボタン横、または上部ツールバー）
   - 「発注済」バッジの表示方法
   - 既に発注書に紐付いた行のチェックボックスを非活性化するロジック
   - 一括発注モーダルの再利用（タブ側の既存実装をほぼそのまま移植）

3. **コンポーネント共有・リファクタリング方針**
   - 一括発注関連のロジックを `hooks/use-bulk-purchase-order.ts` のような形で切り出し可能か
   - チェックボックス・バッジコンポーネントの共通化

4. **修正する具体的なファイル名と差分（diff形式）**

### ステップ3：実装

修正方針を提示し合意した上で、実装してください。

## 制約条件

- **既存の3層構造（発注書 → 仕入伝票 → 原価明細）の動作を壊さないこと**
- **既存の一括発注機能（Task #48）の動作を壊さないこと**
- **既存の仕入先選択ドロップダウン（Task #54）の動作を壊さないこと**
- **既存の発注書詳細・印刷ページの動作を壊さないこと**
- **typecheckエラーを発生させないこと**（`pnpm typecheck` がクリーンに通る）
- **架空のコードや推測で書かない**こと。実際のコードを `view` で確認してから書く
- **削除対象のコードは確実に消す**（コメントアウトで残さない）

## 完了条件

1. プロジェクト詳細ページの「実行予算」タブが、集計カードと「実行予算を編集する」ボタンのみのシンプルな表示になっている
2. 「実行予算を編集する」ボタンをクリックすると `/projects/:id/budgets` に遷移する
3. `/projects/:id/budgets` 画面に以下が統合されている：
   - チェックボックス列
   - 「発注書を作成（N行選択中 / N社）」ボタン
   - 一括発注モーダル（仕入先別グループ・納期/備考入力）
   - 「発注済」バッジ表示
   - 発注済み行のチェックボックス非活性化
   - 「発注済」バッジクリックで発注書詳細ページに遷移
4. 旧「実行予算タブ」のチェックボックス・発注ボタン・鉛筆アイコン経由の行編集モード・削除ボタンが完全に削除されている
5. typecheck がクリーンに通る
6. 既存の各種機能（仕入先選択ドロップダウン・3層構造・発注書詳細・印刷）が引き続き動作する

## 補足

- 「実行予算を編集する」ボタンのデザインは、見積書・請求書など他画面の同種ボタンと統一感を持たせてください
- 一括発注モーダル自体のロジックは Task #48 で完成しているので、**再実装は不要**（既存コンポーネントを使う場所を変えるだけ）
- もし「実行予算タブ」の集計カード（請負金額合計・当初予算合計・実行予算合計・予定利益率）が明細テーブルのデータに依存している場合は、API から直接取得するか、別途集計エンドポイントを用意してください
- このプロンプト③が完了したら、プロジェクト全体の統合テストを再実施するので、その時に発見されたバグは別プロンプトで対応します
